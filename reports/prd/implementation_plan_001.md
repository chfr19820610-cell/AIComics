## 一致性决定

### 1. 数据库: SQLite ✅

**仲裁:** **采纳 PRD 方案 — SQLite**

**理由:**
- PRD 提供了完整的 4 表 SQL DDL（`characters`, `reference_images`, `lora_models`, `project_characters`），含外键、字段约束、索引声明，数据模型设计完备
- 角色系统具有真实的关系需求：角色 ↔ 参考图（一对多）→ 需要外键约束；角色 ↔ LoRA 模型（一对多）→ 需要联表查询；项目 ↔ 角色（多对多）→ 需要关联表
- JSON 文件方案需要手动实现 CRUD、筛选、联表查询、事务一致性、并发写入锁 — 这些是 SQLite 已解决的问题，重新实现是浪费工程资源
- SQLite 零外部依赖（文件级，无需安装服务器进程），与当前项目架构一致
- 补充理由：PRD 中已将 JSON 列为"备选方案"而非推荐方案，PRD 审查 29/30 分未对此提出异议
- **折中:** 保留 JSON 导入/导出功能作为批量角色管理（Phase 3, US-007/A10）的序列化格式，而非存储主方案

**影响评估:** 无架构性变更；PRD 的 SQL DDL 可直接使用。

### 2. Phase 优先级: IP-Adapter 先 ✅

**仲裁:** **采纳 PRD 方案 — Phase 1 = IP-Adapter（零训练方案），Phase 2 = LoRA（强一致性方案）**

**理由:**
- **产品逻辑:** Phase 1 在 2 周内交付可用的端到端价值（角色创建 → 参考图注入 → 75%+ 一致性的漫剧），Phase 2 再提升至 90%+。若先做 LoRA，用户在 4 周内得不到任何交付物
- **工程逻辑:** IP-Adapter 是 ComfyUI 原生节点，核心工作是封装和集成（PRD 估算仅 4 人天）。LoRA 训练管道涉及 Diffusers + PEFT 集成、图像预处理、训练状态反馈 WebSocket、参数推荐引擎（PRD 估算 ~9 人天）— 复杂度远超 IP-Adapter
- **风险控制:** IP-Adapter 先上线可以：
  1. 提前验证角色→参考图→Workflow 集成链路
  2. 收集用户对角色管理系统本身的 UX 反馈
  3. 为 Phase 2 的 LoRA 训练积累角色资产数据
- **竞品对标:** YarnSaga 和 Dashtoon 的核心卖点正是"零训练即可保持一致性"——这是市场的入口级期望，不是可选功能
- **PRD 审查确认:** PRD 审查报告（29/30）未对 Phase 排序提出任何异议

**影响评估:** 架构方案需从 4 周后才见效果调整为 2 周交付可用产品。LoRA 模块的接口设计需向前兼容 IP-Adapter 的数据模型（参考图特征向量字段已预留在 `reference_images.feature_vector` 中）。

### 3. 前端 UI: 分阶段投入 8 人天 ✅

**仲裁:** **采纳 PRD 估算但缩减至 8 人天，分阶段交付**

**理由:**
- 架构方案未估算前端工时是一个缺口，必须补齐。PRD 详细拆分了 5 个 UI 子页面共 11 人天，有明确的颗粒度
- 但对于 Phase 1 (MVP)，11 人天的全量 UI 远超 2 周的交付节奏，需要按阶段重新分配：
  - **Phase 1（~5 人天）:** 角色列表页 + 基本角色创建（非全向导版）+ 基本详情页 — 支持 MVP 角色 CRUD 和参考图上传即可
  - **Phase 2（~2 人天）:** 角色创建向导（完整分步式）+ 效果对比工具
  - **Phase 3（~1 人天）:** 批量管理页面
- 总计 8 人天（较 PRD 原 11 人天减少约 27%），体现了架构方案对前端精简的要求，同时确保每个 Phase 都有可用的 UI

**影响评估:** Phase 1 UI 裁剪为"够用即可"，无需全向导式角色创建 — 先做最小可用页面（form 级别提交），向导放在 Phase 2。后端 API 接口保持不变。
